# CHANGES

Substantive behavioral deviations of pypdfbox vs upstream Apache PDFBox.
Per-release notes go here; trivial naming changes (camelCase → snake_case) are not listed.

## Format

```
- pypdfbox/<path>: <one-line description of deviation>
  upstream: PDFBox <version> <java path>
  reason: <why we deviate>
```

## Project-wide deviations vs upstream

- **No `preflight` module.** Apache PDFBox 4.0 removes Preflight; we follow that decision. PDF/A and PDF/UA validation is performed via external veraPDF / PAC.
- **No commons-logging / log4j.** Python `logging` (stdlib) is used throughout.
- **Method naming.** Java camelCase → Python snake_case across the entire API surface. Semantics unchanged.

## Per-file deviations

- `pypdfbox/io/random_access_read_buffer.py`: wraps `io.BytesIO` instead of reimplementing PDFBox's chunked-list storage. Observable behavior is identical; implementation is C-backed and ~25 lines instead of ~120. Justification: PRD §3.7 (stdlib-first for generic infrastructure).
- `pypdfbox/io/random_access_read_buffered_file.py`: wraps `io.BufferedReader` over a raw file fd. Stdlib provides the read-ahead buffering that upstream's `RandomAccessReadBufferedFile` implements manually. Justification: PRD §3.7.
- `pypdfbox/io/random_access_read_memory_mapped.py`: net-new optional implementation backed by `mmap.mmap`. No upstream counterpart; offered as opt-in for very large files where kernel paging beats userspace buffering. Justification: PRD §3.7 — use stdlib affordances when they fit.
- `pypdfbox/io/random_access_read_view.py`: original slice-view implementation. Mirrors the upstream `RandomAccessReadView` API; storage strategy is direct seek-on-parent rather than upstream's bounded-stream wrapper.
- `pypdfbox/io/scratch_file.py`: backed by `tempfile.SpooledTemporaryFile` (mixed mode), `tempfile.TemporaryFile` (temp-file-only), or `io.BytesIO` (memory-only). Upstream uses page-based scratch storage; we delegate spill-to-disk policy to stdlib. Behavior visible to callers (read/write/seek/clear) is identical. Justification: PRD §3.7. Default spill threshold in MIXED mode without an explicit cap is 16 MiB.

## Backfill — aligned with upstream test expectations

Driven by porting upstream JUnit tests (PRD §12.1):

- `pypdfbox/io/random_access_read.py`: added `read_fully(buf, offset, length)` (raises `EOFError` on premature EOF) and `skip(n)` (clamps to length). Mirrors upstream `RandomAccessRead.readFully` / `skip`.
- `pypdfbox/io/random_access_read_buffer.py`, `_buffered_file.py`, `_memory_mapped.py`, `_view.py`: `seek(negative)` now raises `OSError` (was `ValueError`), `seek(past_end)` clamps to length and leaves the stream at EOF (was: raised `ValueError`). Matches upstream PDFBox semantics; required by ported `seek*` tests.
- `pypdfbox/io/random_access_read_view.py`: removed `start_position + length > parent_length` validation (upstream allows logical views past parent end; reads simply stop at parent EOF). `create_view()` on a view raises `OSError` (upstream forbids).
- `pypdfbox/io/scratch_file.py::ScratchFileBuffer.create_view()` raises `NotImplementedError` (upstream's `UnsupportedOperationException`).
- `pypdfbox/cos/cos_number.py`: new abstract base `COSNumber` (parses string → `COSInteger`/`COSFloat`, handling exponential notation and Java-Long-out-of-range markers). `COSInteger`/`COSFloat` now extend `COSNumber`.
- `pypdfbox/cos/cos_float.py`: values are now clamped to **IEEE-754 single precision** (`float32`) on both `__init__` paths to match Java `float` semantics — required for upstream parity tests on `equals()`/`hashCode()`. `_normalize_negatives` recovers from PDFBOX-2990 / -3369 / -3500 / -4289 misplaced-`-` cases (`--16.33` → `-16.33`, `0.-262` → `-0.262`, `0.00000-33917698` → `-0.0000033917698`); raises `OSError` on unrecoverable double-`-`.
- `pypdfbox/cos/cos_integer.py`: added `is_valid()` / `set_valid()` for PDFBOX-5176 large-integer-out-of-Long-range marking.
- `pypdfbox/cos/cos_string.py`: `parse_hex` raises `OSError` (was `ValueError`) to mirror upstream `IOException`. Added `to_hex_string()`.
- `pypdfbox/cos/cos_stream.py`: `create_raw_input_stream()` raises `OSError` when no body. `create_input_stream(stop_filters=None)` and `create_output_stream(filters=None)` now wire up to `pypdfbox.filter.FilterFactory` and run the `/Filter` chain on read / encode it on close. Encryption-aware stream wrapping (`COSStream.create_view` over an encrypted scratch buffer) is still deferred. `/Predictor` encode + decode are both supported via `FlateDecode` / `LZWDecode` (PNG 10-15 incl. Optimum, TIFF 2).
- `pypdfbox/pdmodel/common/pd_stream.py`: `create_input_stream` and `create_output_stream` now delegate to `COSStream` (single source of truth for filter plumbing). `to_byte_array()` returns `b""` for an empty stream (upstream raises `IOException`); empty `create_input_stream()` returns an empty `BytesIO` rather than raising — `PDStream` is a typed handle that legitimately wraps fresh-and-empty streams. `get_length()` returns `int | None` (was `int`); `None` indicates no `/Length` entry *and* no body.
- `pypdfbox/cos/cos_array.py`: added `remove_object`, `remove_all`, `retain_all`, `grow_to_size`, typed `set_name`/`get_name`/`set_int`/`get_int`/`set_string`/`get_string`, `set_float_array`/`to_float_array`, `to_cos_*_list` converters, factory classmethods `of_cos_names`/`of_cos_strings`/`of_cos_integers`/`of_cos_floats`. All match upstream signatures.
- `pypdfbox/cos/cos_document.py`: added `add_xref_table`, `get_objects_by_type`, `get_linearized_dictionary` (placeholder returning `None` until linearization-hint parsing lands).
- `pypdfbox/cos/cos_name.py`: added single-letter and short-name constants (`A`, `B`, `C`, `D`, `T`, `BE`, `PARAMS`, `FLATE_DECODE`, `ASCII85_DECODE`, `STANDARD_ENCODING`) referenced by ported tests.
- `pypdfbox/pdfparser/base_parser.py::read_name`: a `#` not followed by two hex digits is now kept literally rather than raising `PDFParseError`. Matches upstream `TestBaseParser.testInvalidHexSequence`.

## Filter module + Loader (parallel wave)

- `pypdfbox/filter/ascii85_decode.py`: encode/decode delegate to stdlib `base64.a85encode` / `a85decode` (which already implements PDF's base-85 + `z` shortcut + adobe `<~ ... ~>` framing). Original wrapper handles whitespace pre-cleaning, `~>` terminator stripping, and per-spec error reporting (`OSError`).
- `pypdfbox/filter/flate_decode.py`: `zlib` wrapper. PNG predictors 10-15 + TIFF predictor 2 supported on both decode and encode; predictor 15 (Optimum) picks the per-row filter type that minimises the sum of signed-absolute byte values (RFC 2083 §9.6 heuristic).
- `pypdfbox/filter/ascii_hex_decode.py`: `binascii` wrapper. Whitespace ignored, odd-digit pad with `0`, terminates on `>`.
- `pypdfbox/filter/run_length_decode.py`: ported line-for-line so encoded bytes match upstream byte-identical.
- `pypdfbox/filter/lzw_decode.py`: PDF-flavored LZW (9-12 bit codes, MSB-first, EarlyChange honored on decode; encoder always emits EarlyChange=1 with the PDFBOX-1977 final-EOD-width fix).
- `pypdfbox/filter/ccitt_fax_decode.py`: T.4 (Group 3, K=0 / K>0) and T.6 (Group 4, K<0) decode by wrapping the encoded strip in a synthetic little-endian TIFF and handing it to libtiff via Pillow. `/Columns`, `/Rows`, `/K`, `/BlackIs1`, `/EncodedByteAlign` honored; `/EndOfLine`, `/EndOfBlock`, `/DamagedRowsBeforeError` accepted but inert (libtiff handles framing). `/Rows` omitted → libtiff fills a generous over-allocated height with zero-padding past actual end-of-data; callers should supply `/Rows` from the image XObject's `/Height` for a tight buffer. Decode-only (no encoder use case yet — `encode()` raises `NotImplementedError`). Output is raw bit-packed scanlines (rows padded to whole bytes, MSB-first), the same shape PDImageXObject's decoded body expects.
- `pypdfbox/filter/jbig2_decode.py`: `/JBIG2Decode` (ISO 32000-1 §7.4.7) decode via the Rust-backed `jbig2-parser` library (MIT, newly approved dependency). `/JBIG2Globals` from `/DecodeParms` (single dict or array indexed by filter position; `/DP` short-form alias also accepted) is decoded through its own `/Filter` chain via `COSStream.to_byte_array()` and prepended to the per-image segments before handoff to `jbig2_parser.parse_jbig2`. The parser returns a PNG buffer; we re-decode it through Pillow to recover raw bilevel sample bytes. JBIG2 is bilevel by ITU-T T.88 definition, so `DecodeResult.parameters` is always populated with `/BitsPerComponent = 1` and `/ColorComponents = 1` plus the codestream's intrinsic `/Width` and `/Height`. PDF spec defines no short-name abbreviation for `/JBIG2Decode` — only the long name is registered. Decode-only (`encode()` raises `NotImplementedError`); a malformed `/JBIG2Globals` value of the wrong COS type is silently ignored rather than crashing the image decode.
- `pypdfbox/filter/_predictor.py` (new): shared PNG/TIFF predictor encode (`predict`) + decode (`unpredict`) helpers, used by both `FlateDecode` and `LZWDecode`. Replaces the previously-duplicated inline `_unpredict` blocks in each filter. Supports predictors 1, 2, 10-15 across 1/2/4/8/16-bit components and any colors/columns combo.
- `pypdfbox/cos/cos_document.py`: optional `source: RandomAccessRead | None` parameter / instance attribute so `Loader` can attach the file handle it created and have `doc.close()` close it. Caller-supplied sources stay caller-owned.
- `pypdfbox/loader.py`: top-level `Loader.load_pdf()` accepts path / `bytes` / `bytearray` / `memoryview` / `BinaryIO` / `RandomAccessRead`. Mirrors `org.apache.pdfbox.Loader.loadPDF()`. Encryption / password / `MemoryUsageSetting` parameters deferred until those features ship.
- `pypdfbox/__init__.py`: exports `Loader` so `from pypdfbox import Loader` works (matches PRD §7).
- `pypdfbox/pdfparser/pdf_stream_parser.py`: extends `COSParser`. Surfaces an `Operator` value type carrying the operator name plus optional `image_parameters` / `image_data` for inline-image (`BI` ... `ID` ... `EI`) sequences. Lenient PDFBox quirks preserved: stray `]` returns `COSNull`, isolated `+` returns `COSNull`, internal `-` in numbers dropped (PDFBOX-4064), Type-3 `d0`/`d1` operator suffix accepted.

## pdfwriter cluster #1 (full-save, traditional xref)

- `pypdfbox/pdfwriter/cos_writer.py`: cluster #1 covers full-save mode only. The following upstream paths are stubbed and raise `NotImplementedError` until later clusters land:
  - `incremental=True` constructor flag → cluster #2 (incremental save).
  - Xref-stream output (`writeXrefStream` and friends) → cluster #3.
  - Object-stream packing (`COSWriterObjectStream`) → cluster #3.
  - Encrypted documents (`is_encrypted()` → `True`) → security cluster.
  - Signatures (`SignatureInterface`, `/ByteRange` ranges) → cluster #2.
- `pypdfbox/pdfwriter/cos_writer.py`: `write(document)` accepts a `COSDocument` directly. Upstream's overloads also take `PDDocument` / `FDFDocument`; those wrappers are not yet ported (`PDDocument` → PRD §6.6 cluster).
- `pypdfbox/pdfwriter/cos_writer.py`: `/ID` synthesis uses `hashlib.sha256(time_ns + secrets.token_bytes(16))[:16]` instead of upstream's `Long.toString(idTime) + info-dict-values` digest. Same shape (two 16-byte hex strings) and same purpose; we don't have a `PDDocument.getDocumentId()` to thread in yet.

## pdfwriter cluster #2 (incremental save)

- `pypdfbox/pdfwriter/cos_writer.py`: `COSWriter(..., incremental=True, incremental_input=...)` now ships. The writer copies the source bytes to the output verbatim and appends only objects whose resolved value (or its `COSObject` wrapper) carries `is_needs_to_be_updated()=True`. The appended xref section lists only the changed/new objects, the trailer carries `/Prev = old_startxref`, `/Size = max_obj_num + 1`, and preserves `/ID` exactly as the source had it. Mirrors upstream `doWriteIncrement` + `prepareIncrement` + `doWriteXRefInc`.
- `pypdfbox/pdfwriter/cos_writer.py`: digital-signature re-signing is **not** implemented. When a source signature dict (`/Type /Sig` or `/DocTimeStamp`) contains a `/ByteRange [0 0 0 0]` placeholder (or any byterange whose third entry doesn't extend past the original file end), `save_incremental` raises `NotImplementedError` rather than silently corrupt the signature. The actual digest pipeline lands with the security cluster.
- `pypdfbox/pdfwriter/cos_writer.py`: `save_incremental` on a document with **no** dirty objects produces an output **byte-for-byte identical to the source**. Upstream's behaviour matches: nothing extra is appended. Unlike full-save mode, no `/ID` synthesis happens in incremental mode — the source's `/ID` array (if any) stays exactly as it was.
- `pypdfbox/pdfwriter/cos_writer.py`: an in-memory `BytesIO` increment buffer accumulates the appended objects + xref + trailer; the real output sink only sees source-bytes followed by buffer-contents at the very end. Mirrors upstream's `ByteArrayOutputStream` → `incrementalOutput` pipeline. The buffer is initialised with `position=0`; absolute byte offsets in the new xref are computed as `source_length + buffer_offset` at emit time (upstream cheats by seeding `position` with `inputData.length()` on the `COSStandardOutputStream`; functionally identical, easier to test in isolation).
- `pypdfbox/pdfwriter/cos_writer.py`: a CRLF separator is emitted at the very start of the increment buffer when there is at least one dirty object. This matches upstream's `getStandardOutput().writeCRLF()` at the top of `visitFromDocument` in incremental mode and guarantees an unambiguous boundary between the source's trailing `%%EOF` and the appended block. When nothing is dirty the separator is **not** emitted (preserving the byte-for-byte-identical contract).
- `pypdfbox/cos/cos_document.py`: added `get_source()` (read-only accessor for the parser-attached `RandomAccessRead`), `get_start_xref()` / `set_start_xref(offset)` (the offset of the trailing xref section, used by the incremental writer for `/Prev`).
- `pypdfbox/pdfparser/pdf_parser.py`: after parsing the trailing `startxref` value, calls `document.set_start_xref(...)` so the incremental writer can find it later.

## pdfwriter cluster #3 (xref-stream + object-stream output)

- `pypdfbox/pdfwriter/cos_writer.py`: `COSWriter(..., xref_stream=True)` (with `set_xref_stream(b)` / `is_xref_stream_output()` toggles) emits a single `/Type /XRef` indirect-object stream in place of the traditional `xref` keyword + `trailer` keyword pair (PDF 32000-1 §7.5.8). Trailer entries (`/Root`, `/Info`, `/Encrypt`, `/ID`) are folded into the xref-stream dictionary as required by §7.5.8.4. `/W` field widths are computed dynamically (`w1=1`, `w2=ceil_log256(max_offset)`, `w3=2`); `/Index` always lists the contiguous object-number ranges. The xref-stream object goes through the regular `visit_from_stream` path so its body is FlateDecode-compressed and — when an active security handler is staged — encrypted with its own per-object key. Default remains the traditional xref table.
- `pypdfbox/pdfwriter/cos_writer.py`: `COSWriter(..., object_stream=True)` (with `set_object_stream(b)` / `is_object_stream_output()` toggles) packs eligible non-stream indirect objects into one or more `/Type /ObjStm` streams (PDF 32000-1 §7.5.7), capped at 100 objects per ObjStm to mirror upstream PDFBox's `COSWriterObjectStream` ceiling. Streams, the `/Encrypt` dictionary, and signature dictionaries (`/Type /Sig`, `/Type /DocTimeStamp`) are excluded by spec. Type-2 xref entries are emitted in the surrounding xref stream; the regular indirect-object frame for a packed object is suppressed. Object-stream packing requires xref-stream output (the cluster does not auto-promote — callers must set both flags).
- `pypdfbox/pdfwriter/cos_writer.py`: hybrid xref tables (`/XRefStm` pointing back at a traditional xref) — supported by upstream for backward compatibility with PDF-1.4-only readers — are not yet emitted. Callers that need that fallback should save with `xref_stream=False`.

- `pypdfbox/contentstream/operator.py`: operands list is stored on the `Operator` instance (`get_operands()` / `set_operands()`). Upstream keeps the operand stack on `PDFStreamEngine` / `PDFStreamParser`; pypdfbox attaches operands directly to the `Operator` returned by the parser as a convenience for token consumers (the parser already does this — see `pdf_stream_parser.py`). The cached singletons returned by `Operator.get_operator(name)` for ordinary operators must therefore be treated as flyweights — do not mutate operands on a cached instance you did not just receive from the parser.
- `pypdfbox/contentstream/operator.py`: cache uses a plain `dict` plus a `threading.Lock` instead of upstream `ConcurrentHashMap.putIfAbsent`. Functionally identical under the GIL.
- `pypdfbox/contentstream/pd_content_stream.py`: `get_matrix()` is typed `Any` rather than `Matrix` because `pypdfbox.util.Matrix` is not yet ported (it lands with the rendering cluster). Subclasses may return a `COSArray` (the on-disk form) until then.

## pdmodel cluster #3 (PDStream + XObject family)

- `pypdfbox/pdmodel/common/pd_stream.py`: stream construction/output with `filters` records the `/Filter` chain but stores caller-supplied bytes as-is. Upstream writes through `COSStream.createOutputStream(filters)` and therefore encodes on write; pypdfbox defers encode-on-write until the filter encoding surface is widened. Decoding existing filtered bytes is supported through `FilterFactory`.
- `pypdfbox/pdmodel/graphics/image/pd_image_x_object.py`: image decoding, color-space wrapping, array color spaces, and rendered image conversion are deferred to rendering / graphics-color clusters. ## PDFunctionType4 PostScript eval

- `pypdfbox/pdmodel/common/function/pd_function_type4.py`: `eval(input)` now ships per PDF 32000-1 §7.10.5. Body bytes are read through `COSStream.create_input_stream` (so `/Filter` chains decode), tokenised whitespace+brace, parsed into a nested instruction sequence (each `{...}` block becomes a Python `list`), and run through a minimal stack machine. Operator coverage: arithmetic (add/sub/mul/div/idiv/mod/neg/abs/ceiling/floor/round/truncate/sqrt/sin/cos/atan/exp/ln/log/cvi/cvr), stack (dup/exch/pop/copy/index/roll), boolean (eq/ne/lt/le/gt/ge/and/or/xor/not/bitshift/true/false), conditional (if/ifelse). `def`, `forall`, `for`, dictionaries, and other operators not allowed in Type 4 are intentionally rejected (raise `OSError`). Numeric stability uses Python `float`; booleans live on the same stack as numbers per spec and are coerced to `1.0`/`0.0` on the way out for `/Range` clipping. Stack underflow / type mismatch / unknown operator all surface as `OSError` (mirrors upstream `IOException`). Deferred: instruction-sequence caching (we re-parse on every `eval` call), richer error-recovery semantics, and cross-checking numeric edge cases against Java's IEEE-754 single-precision rounding.

## PDSignature write-side pipeline

- `pypdfbox/pdmodel/pd_document.py`: `add_signature(sig, signature_interface, options=None)` now ships. Accepts a typed `PDSignature` plus a `SignatureInterface` callback. Wires the signature dict into `/AcroForm /Fields` inside an invisible signature widget (`/T = "Signature{N}"`, `/Rect = [0 0 0 0]`, `/Subtype /Widget`, `/F = 132`), anchors the widget to the first page's `/Annots`, sets `/SigFlags = 3` (SignaturesExist | AppendOnly), and stages the signer for the next `save_incremental`. Does **not** create a new field name slot when one already exists — it picks the next free `Signature{N}`.
- `pypdfbox/pdmodel/pd_document.py`: `save_incremental` now drives the signing pipeline when `add_signature(...)` has staged a pending signature. Pre-flight phase reserves a `/Contents <0×16384>` placeholder (8192 raw bytes, headroom for typical RSA-2048 PKCS#7 detached blobs + chain) and a `/ByteRange [0 9999999999 9999999999 9999999999]` placeholder, runs the writer into a `BytesIO`, locates the placeholder, computes the real `/ByteRange [0 X Y Z]` bracketing everything outside the hex `<…>` (the angle brackets themselves stay inside the hashed range per ISO 32000-1 §12.8.1), splices the real ByteRange in place (trailing-space pad keeps width identical), invokes `signature_interface.sign(io.BytesIO(bracketed_bytes))` to produce the DER PKCS#7 blob, and patches the hex-encoded blob into the `/Contents` slot (zero-pads to placeholder length). The bracketed-bytes math is `(X-0) + (Z-(Y-1)) = file_size - hex_placeholder_len`. Final bytes are drained to the user-supplied `target` (path / `BinaryIO` / `RandomAccessWrite`).
- `pypdfbox/pdmodel/pd_document.py`: `save_incremental_for_external_signing(output) -> ExternalSigningSupport` runs the same pipeline minus the `signature_interface.sign` call. Returns an `ExternalSigningSupport` handle exposing `get_content() -> bytes` (the bracketed bytes the caller hashes + signs, e.g. via an HSM) and `set_signature(pkcs7_der: bytes) -> None` (splices the blob in, drains to `output`). `add_signature` may be called with `signature_interface=None` to use this path.
- `pypdfbox/pdmodel/pd_document.py`: new `ExternalSigningSupport` helper class (no upstream-specific equivalent in pypdfbox — the upstream `org.apache.pdfbox.pdmodel.interactive.digitalsignature.ExternalSigningSupport` is mirrored here as a small dataclass-ish shim).

## Rendering cluster #1 — Pillow + aggdraw page rasteriser

- `pypdfbox/rendering/pdf_renderer.py` (NEW): `PDFRenderer(document)` mirrors `org.apache.pdfbox.rendering.PDFRenderer`. Subclasses `PDFStreamEngine` and folds the page-drawing logic that upstream splits into `PageDrawer` directly into operator handlers. Backend is **Pillow + aggdraw** — aggdraw provides anti-aliased AGG-quality stroking, native cubic Bezier path segments, and PIL-style `(a, b, c, d, e, f)` affine transforms via `Draw.settransform(...)`. Operators implemented: path construction (`m`, `l`, `c`, `v`, `y`, `re`, `h`), painting (`S`, `s`, `f`/`F`, `f*`, `B`, `B*`, `b`, `b*`, `n`), graphics state (`q`/`Q` push/pop CTM + colour + line width), transform (`cm`), colour (`RG`/`rg`/`G`/`g`/`K`/`k` for DeviceRGB/DeviceGray/DeviceCMYK), line width (`w`), and image XObject `Do` for `/Subtype /Image` (JPEGs via `/Filter /DCTDecode` open through PIL after stop-filtering at DCTDecode; raw 8-bpc DeviceRGB / DeviceGray rasters via `Image.frombytes`). Even-odd fill rule (`f*`/`B*`/`b*`) is implemented by flattening Beziers and XOR-merging per-subpath PIL polygon masks (aggdraw's path fill is non-zero only); the resulting binary mask loses sub-pixel AA on the outer edge, a documented limitation. Initial CTM bakes the PDF→PIL Y-axis flip plus DPI scale plus media-box origin offset into a single 6-tuple. Per-render aggdraw `Draw` is rebuilt after any direct PIL paste so its internal pixel buffer never overwrites pasted-mutation regions on `flush()`. Deferred to text/font cluster: text operators (`BT`/`ET`/`Tj`/`TJ`/`'`/`"`) are silently skipped — real glyph rendering needs the font cluster's `fontTools` glyph paths. Deferred entirely (silent skip): shadings, patterns, transparency groups, soft masks, blend modes, line dash/cap/join, clipping paths, form XObjects, inline images. `render_image(page_index, scale=1.0)` and `render_image_with_dpi(page_index, dpi=72.0)` mirror upstream's two entry points.

## Rendering cluster #2 — text + form XObjects + clip + inline images

- `pypdfbox/rendering/pdf_renderer.py`: extended with four operator families.
  - **Text**: `BT`/`ET`/`Tf`/`Tc`/`Tw`/`TL`/`Tz`/`Ts`/`Td`/`TD`/`Tm`/`T*`/`Tj`/`TJ`/`'`/`"`. The `_GState` dataclass now carries text state (font, font size, text matrix, text line matrix, char spacing, word spacing, leading, rise, horizontal scaling). Glyph outlines for embedded TrueType fonts come through fontTools (`tt.getGlyphSet()[name].draw(pen)`) into a custom `_AggdrawPathPen` that scales coordinates to unit-em and emits aggdraw `Path.curveto`/`lineto`/`moveto`/`close`. Quadratic TT segments are converted to exact cubic Beziers via the standard 1/3 control-point elevation. `qCurveTo`'s implicit-on-curve walk follows fontTools' TT convention (consecutive off-curves get a halfway implicit on-curve). `PDType0Font` with a `CIDFontType2` descendant pulls the descendant's `/FontDescriptor /FontFile2` and reuses the same path. `PDType1Font` / `PDType3Font` / Standard 14 (no embedded program) silently fall back to a faint placeholder rectangle covering the glyph's advance × ~0.7 em — non-fatal, no crash. Per-font lookups are cached on `(id(resources), font_key)` so `Tf` doesn't re-parse the embedded TTF on every text run.
  - **Form XObjects** (`Do` for `/Subtype /Form`): saves the GS, post-multiplies the form's `/Matrix` onto the CTM, intersects the form's `/BBox` into the current clip mask, swaps `_resources` for the form's own `/Resources` if present, then re-feeds the form's content-stream bytes through the engine's dispatch loop via a small `_process_form_bytes` helper (a thin wrapper around `PDFStreamParser` over a `RandomAccessReadBuffer` — the form is not yet a `PDContentStream` subclass).
  - **Clip** (`W` / `W*`): a `_pending_clip` flag is set; the next `n` (or any path-paint operator) builds a polygon mask in device space (Beziers flattened by 16-step sampling, even-odd via `ImageChops.logical_xor` per subpath, non-zero via per-subpath polygon union), intersects it with any existing GS clip via `ImageChops.multiply`, and stores the result on `_GState.clip_mask`. Subsequent paint operators render into a transparent RGBA layer and composite back through `clip_mask`. Image pastes mask their bbox by `(rect_mask × clip_mask)`.
  - **Inline images** (`BI` … `ID` … `EI`): the parser already collapses the triplet into a single `BI` operator carrying both `image_parameters` (the dict) and `image_data` (the raw bytes). `_op_inline_image` synthesises a transient PIL image from those, recognising the abbreviated keys per PDF spec §8.9.7 Table 92 (`W`/`H`/`CS`/`BPC`/`F`) and the abbreviated colour-space + filter names (`G`/`RGB`/`CMYK`, `DCT`/`JPX`/`Fl`/`A85`/`AHx`/`CCF`/`LZW`/`RL`), then routes through the same `_paste_image` path as `Do`-image.
  - **Deferred (silent skip)**: shadings, patterns, transparency groups, soft masks, blend modes, line dash/cap/join, `Tr` text rendering modes (clipping/stroke), Type1/Type3/Standard 14 glyph outlines (placeholder rectangle).

## Rendering cluster #3 — Type1 / Type1C (CFF) text rasterisation

- `pypdfbox/rendering/pdf_renderer.py`: glyph dispatch in `_show_string` / `_draw_glyph` now branches on font kind. The TTF branch (cluster #2) is preserved unchanged. New branch: when the resolved PDFont is a `PDType1Font` (PFB-backed embedded Type 1) or `PDType1CFont` (CFF / Type1C — `/FontFile3 /Subtype /Type1C`), the renderer pulls the per-glyph command tuples from `font.get_glyph_path(code)` (Wave 17 fontTools-backed APIs), feeds them into a small `_build_aggdraw_path_from_commands` helper that converts (`moveto`, `lineto`, `curveto`, `closepath`) into `aggdraw.Path` segments scaled by `1 / units_per_em`, and reuses the same `_fill_aggdraw_path` pipeline as the TTF branch (clip-mask-aware fill onto the device CTM). `units_per_em` is sourced from `Type1Font.units_per_em` / `CFFFont.units_per_em` (typically 1000 for the canonical Type 1 font matrix `[0.001 0 0 0.001 0 0]`). Subclasses of `PDType1Font` (`PDMMType1Font`, `PDType1CFont`) are matched via isinstance and ride the same code path.
- **Standard 14 substitution remains deferred.** When a font names a Standard 14 family (Helvetica, Times-Roman, Courier, Symbol, ZapfDingbats, …) but ships *no* embedded program (no `/FontFile`, no `/FontFile3`), `get_glyph_path` returns `[]` and we still emit the cluster #2 placeholder rectangle. To make the gap explicit, the renderer now logs **once per font** at `DEBUG` (`"Standard 14 ... using placeholder rectangle (Liberation TTF substitution not yet bundled — see CHANGES.md)"`) — tracked on `_warned_standard14_fonts: set[id(font)]`. Two viable substitution paths remain on the table for a future cluster: (a) bundling Liberation TTFs from PDFBox's resources (SIL OFL — permissive but the OFL font-rename clause adds NOTICE complexity), or (b) deferring further until users hit the gap with a real corpus. Trade-off: Liberation gives PDFBox-parity Standard-14 rendering at the cost of ~2 MB of bundled fonts and an OFL attribution row in NOTICE; deferring keeps the wheel lean but means Helvetica/Times/Courier render as faint outline boxes.
